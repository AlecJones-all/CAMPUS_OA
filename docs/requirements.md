# 校园 OA 需求分析说明书

## 1. 项目背景

校园大量业务仍依赖纸质材料、微信群通知、Excel 台账和线下签字，存在流程不透明、统计困难、责任追踪困难和材料归档分散的问题。校园 OA 系统的一期目标是建设统一的信息化平台，将学生事务、教学教务、科研管理、行政后勤等高频流程纳入同一套权限、流程和数据体系。

## 2. 建设目标

- 建设统一入口和工作台，集中处理待办、已办、通知和报表
- 建设统一组织、角色、权限、流程、附件和消息底座
- 覆盖一期高频流程，形成标准化线上闭环
- 保留面向后续移动端、单点登录和复杂流程引擎的扩展能力

## 技术基线

- 前端：`Vue 3 + Vite + TypeScript + Element Plus`
- 前端配套：`Pinia + Vue Router`
- 后端：`Spring Boot`
- 后端配套：`Spring Security + MyBatis-Plus`
- 数据库：`MySQL`
- 文档：`Markdown`

## 3. 建设范围

### 3.1 一期范围

- 基础平台能力
- 学生事务
- 教学教务
- 科研项目
- 行政后勤
- 统计报表与预警

### 3.2 二期预留

- 小程序与移动端
- 学校统一身份认证
- 短信、邮件、企业微信等消息推送
- 复杂 BPM 编排
- 对象存储与电子签章

## 6. 典型业务流程

### 6.1 请假与销假

1. 学生发起请假申请并上传证明材料
2. 班主任或辅导员审批
3. 根据时长或类型进入学院审批
4. 审批通过后形成请假台账
5. 学生返校后提交销假申请
6. 班主任确认销假，流程办结

### 6.2 实习材料管理

1. 学生提交实习协议、单位证明、周报或总结
2. 班主任批量查看材料完整性
3. 对缺失、异常材料退回补交
4. 学院汇总统计提交率和异常情况
5. 教务处查看备案与报表

### 6.3 异常学生材料管理

1. 班主任创建异常学生跟踪记录
2. 录入异常类型、原因、谈话记录和附件
3. 学院复核并安排干预
4. 系统按周期提醒跟踪
5. 闭环后归档

### 6.4 课程标准评审

1. 任课教师提交课程标准文档
2. 教研室分配评审人
3. 评审人给出意见与评分
4. 教研室汇总处理结果
5. 学院备案，教务处抽查

### 6.5 课题申报评审

1. 教师填写课题申报表并提交附件
2. 学院审核申报资格和材料完整性
3. 科技处分配专家评审
4. 专家评分和意见汇总
5. 形成立项结果并归档

### 6.6 维修工单

1. 学生或教师提交维修申请
2. 管理员派单给维修人员
3. 维修人员处理并上传结果
4. 申请人验收和评价
5. 完成闭环统计

## 7. 功能需求

### 7.1 工作台

- 待办、已办、我发起、抄送
- 公告和消息提醒
- 业务快捷入口
- 超时预警提示

### 7.2 通用流程中心

- 支持按业务类型配置流程模板
- 支持固定审批链和基于条件的分支审批
- 支持撤回、退回、重新提交
- 支持查看流程轨迹和审批意见

### 7.3 统一附件中心

- 上传、下载、预览
- 文件分类与业务绑定
- 审计记录
- 批量导出预留

### 7.4 报表中心

- 提交率
- 审批时效
- 超时工单
- 学院、专业、班级维度统计

## 8. 非功能需求

- 安全：鉴权、权限控制、操作日志、敏感数据留痕
- 性能：常用列表和表单在校内日常使用场景下响应稳定
- 可维护性：统一编码规范、模块化设计、接口文档清晰
- 可扩展性：支持新增流程和新增业务域
- 易用性：中后台操作简单、流程状态清晰、错误提示明确

## 9. 数据设计原则

- 组织、用户、角色、权限独立建模
- 流程定义与流程实例分离
- 业务状态与流程状态分离
- 附件统一管理，不在业务表中直接存文件信息
- 审批意见不覆盖，必须保留完整历史
- 支持按组织和时间维度做统计分析

## 10. 验收标准

- 至少 1 套基础平台能力可运行
- 至少 2 个业务域可直接接入流程底座
- 一期关键流程具备提交、审批、退回、查询、统计能力
- 角色权限矩阵明确且无明显越权风险
- 文档、数据库、前后端结构可以支撑后续继续开发

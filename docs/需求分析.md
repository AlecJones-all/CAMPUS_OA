# 校园 OA 系统需求分析

## 1. 项目背景

当前校园内大量管理事务仍以纸质表单、线下签字、微信群通知、电话沟通和 Excel 台账为主，存在以下共性问题：

- 流程分散，发起、审批、反馈、归档之间缺乏统一入口
- 审批链不透明，申请人无法实时了解流程进度
- 材料收集依赖人工催交，附件版本混乱，难以统一归档
- 统计工作依赖人工汇总，耗时长且容易出错
- 不同部门、学院、角色之间数据边界不清晰，存在越权风险
- 超时流程、异常学生、维修积压等问题缺少及时预警
- 同一类业务在不同学院或部门执行标准不统一，重复劳动较多

为了提高校园治理和日常办公效率，需要建设一套统一的校园 OA 系统，将学生事务、教学教务、科研管理、行政后勤等高频业务逐步信息化，实现统一登录、统一权限、统一流程、统一附件、统一消息和统一统计。

本系统面向学生、班主任、辅导员、任课教师、教研室、学院、教务处、科技处、学工处、后勤部门、宿管部门、资产管理部门和系统管理员等角色，服务于校内多部门协同办公和业务闭环管理。

## 2. 建设目标

### 2.1 总体目标

- 建设覆盖校园多业务域的统一 OA 平台
- 建设统一工作台，整合待办、已办、我发起、抄送、通知和报表
- 建设统一组织、用户、角色、菜单、权限和数据权限体系
- 建设统一流程中心，支持常见审批流和留痕管理
- 建设统一附件中心，支持材料上传、下载、预览和归档
- 建设统一消息中心，实现通知、待办提醒和流程反馈
- 建设统一统计分析能力，支持按学院、专业、班级、时间等维度汇总

### 2.2 一期目标

- 完成校园 OA 的需求分析、开发计划和数据库设计草案
- 完成前后端基础工程和数据库初始化结构
- 实现系统底座，包括登录认证、用户组织、角色权限、菜单、字典、附件、消息、日志
- 实现流程中心，包括发起、审批、退回、撤回、转交、轨迹查看、待办已办
- 优先落地高频业务流程，形成可演示、可验证的业务闭环

### 2.3 二期预留目标

- 接入学校统一身份认证
- 支持移动端或小程序审批
- 接入短信、邮件、企业微信等消息渠道
- 支持更复杂的流程编排与可视化配置
- 支持对象存储、电子签章和更完善的归档能力

## 技术基线说明

为保证需求文档、代码实现和后续开发路线一致，项目当前统一采用以下技术基线：

- 前端：`Vue 3 + Vite + TypeScript + Element Plus`
- 前端配套：`Pinia + Vue Router`
- 后端：`Spring Boot`
- 后端配套：`Spring Security + MyBatis-Plus`
- 数据库：`MySQL`
- 文档：`Markdown`

## 3. 用户角色

系统涉及的主要角色如下：

### 3.1 学生

- 发起学生事务类申请
- 提交材料、补交材料
- 查看审批进度和处理结果
- 接收通知和回执要求

### 3.2 班主任 / 辅导员

- 审核学生请假、销假、证明、奖助学金等事务
- 收集和审核学生实习材料
- 建立和跟踪异常学生档案
- 查看班级维度统计数据

### 3.3 任课教师

- 发起调课、停课、补课等教学类申请
- 提交课程标准、教学大纲、教案等教学资料
- 发起课题申报、中期检查、结题等科研类流程

### 3.4 教研室主任 / 教研室秘书

- 审核课程标准、教材征订、教学资料
- 管理教研室范围内的教学流程
- 组织专业性评审

### 3.5 学院教学秘书 / 学院管理人员

- 汇总学院内教学教务数据
- 审核学院层面的业务申请
- 组织学院内业务统计和材料归档

### 3.6 学院领导

- 审批学院层面的重点业务
- 查看学院范围内统计报表和流程运行情况

### 3.7 教务处

- 审核教学教务类关键流程
- 配置教学管理规则
- 维护全校教学类统计口径

### 3.8 科技处

- 管理课题申报、评审、中期检查、结题和科研成果
- 配置科研类评审与归档规则

### 3.9 学工处

- 管理奖助学金、困难认定、学生事务统计
- 查看学生事务全校范围数据

### 3.10 后勤 / 宿管 / 资产管理员 / 实验室管理员

- 处理会议室、物资借用、宿舍维修、资产报修、实验室安全等业务
- 管理相关资源和工单状态

### 3.11 评审专家

- 参与课题申报、课程标准等业务的评审
- 查看分配给本人的评审任务并填写评审意见

### 3.12 系统管理员

- 维护用户、组织、角色、菜单、字典、参数
- 配置流程模板和系统基础数据
- 查看操作日志、登录日志和系统运行状态

## 5. 详细业务流程

本节列出一期优先建设的核心业务流程及其主要处理逻辑。

### 5.1 学生请假与销假流程

#### 目标

- 实现学生请假线上审批和返校销假闭环

#### 流程说明

1. 学生填写请假申请并上传证明材料
2. 系统校验申请信息完整性和时间范围
3. 班主任或辅导员进行初审
4. 根据请假类型、时长或特殊情况决定是否进入学院审批
5. 审批通过后生成请假记录，并向相关人员发送通知
6. 学生返校后提交销假申请
7. 班主任或辅导员确认销假，流程办结

#### 关键控制点

- 请假时间不能与既有未销假记录冲突
- 退回后允许学生补充材料重新提交
- 请假与销假需形成完整台账

### 5.2 实习材料提交流程

#### 目标

- 实现学生实习材料统一收集、审核和归档

#### 流程说明

1. 学生按材料类型上传实习协议、单位证明、周报、总结等
2. 班主任查看班级提交进度并审核材料完整性
3. 对缺失或不合格材料进行退回补交
4. 学院对整体提交率和异常情况进行汇总
5. 教务处进行备案或抽查

#### 关键控制点

- 材料需按学期、学生、材料类型分类管理
- 支持班主任批量查看和催交
- 所有材料要与学生和业务状态绑定

### 5.3 异常学生材料管理流程

#### 目标

- 对学业、心理、违纪、就业等异常学生建立跟踪档案

#### 流程说明

1. 班主任或辅导员发起异常学生记录
2. 填写异常类型、异常级别、原因、谈话记录和处理措施
3. 上传相关证明材料或谈话纪要
4. 学院复核处理方案
5. 系统按设定周期提醒继续跟踪
6. 达到闭环条件后归档

#### 关键控制点

- 不同异常类型需要不同跟踪周期
- 重大异常需增加学院领导审批或确认
- 全过程必须保留历史记录

### 5.4 教师调课流程

#### 目标

- 统一调课审批、通知和冲突控制

#### 流程说明

1. 任课教师发起调课申请
2. 填写课程、原时间、调整后时间、教室、原因等信息
3. 教研室审核合理性
4. 学院审核学院层面安排
5. 教务处确认并发布变更通知
6. 学生接收调课通知

#### 关键控制点

- 必须校验时间和教室资源冲突
- 调课结果需要同步到通知中心

### 5.5 课程标准评审流程

#### 目标

- 统一课程标准提交、评审、备案和归档

#### 流程说明

1. 任课教师上传课程标准文件并填写课程信息
2. 一期由教务处审核角色接收待办并进行评审
3. 审核人填写意见和评分
4. 形成评审结论
5. 学院备案
6. 后续可细化为教研室分配评审与汇总

#### 关键控制点

- 文档版本需可追踪
- 评审意见不可覆盖历史记录
- 一期先采用教务处审核角色跑通主链，后续再细化教研室角色与分工

### 5.6 课题申报流程

#### 目标

- 建立科研项目从申报到评审的统一流程

#### 流程说明

1. 教师填写课题申报表并上传附件
2. 学院审核申报资格和材料完整性
3. 科技处分配专家评审
4. 专家在线给出评分和意见
5. 科技处汇总结果并形成结论
6. 项目立项或退回修改

#### 关键控制点

- 支持多专家评审
- 支持评审意见汇总和结果归档
- 支持后续中期检查和结题关联

### 5.7 会议室申请流程

#### 目标

- 统一会议室预约和资源排期

#### 流程说明

1. 申请人填写会议主题、时间、人数、设备需求
2. 系统校验时间冲突
3. 管理员或归属部门审批
4. 审批通过后锁定会议室时段
5. 会后保留使用记录

#### 关键控制点

- 必须防止重复占用
- 需要支持不同会议室不同审批规则

### 5.8 宿舍维修 / 资产报修流程

#### 目标

- 建立报修工单、派单、处理、验收的标准闭环

#### 流程说明

1. 学生或教师发起报修申请
2. 填写地点、报修类型、问题描述并上传图片
3. 管理员受理并派单
4. 维修人员处理并上传处理结果
5. 申请人确认验收和评价
6. 工单办结并纳入统计

#### 关键控制点

- 紧急工单需支持优先级
- 维修过程需留痕
- 支持按区域、类型和时效进行统计

## 7. 非功能性需求

### 7.1 安全性需求

- 用户必须登录后才能访问受控资源
- 菜单、按钮、接口和数据必须多层权限控制
- 审批、导出、配置修改等关键操作必须记录日志
- 重要业务数据应具备审计能力

### 7.2 性能需求

- 常用页面和列表在日常办公场景下响应稳定
- 常规查询、审批、上传附件等操作应具备可接受的响应速度
- 支持后续扩展更大业务量而不推倒重来

### 7.3 可维护性需求

- 系统采用模块化设计
- 前后端结构清晰，公共能力可复用
- 文档、数据库、接口设计保持同步维护

### 7.4 可扩展性需求

- 支持新增业务类型
- 支持新增流程模板和审批规则
- 支持后续接入移动端、统一认证和对象存储

### 7.5 易用性需求

- 页面结构清晰
- 表单字段说明明确
- 审批状态直观
- 错误提示和引导友好

## 8. 权限需求

### 8.1 权限模型

系统权限分为三层：

- 菜单权限：决定用户是否可以进入某功能模块
- 按钮权限：决定用户是否可以执行新增、编辑、审批、导出等操作
- 数据权限：决定用户可以查看和处理哪些范围的数据

### 8.2 数据权限范围

- 学生：仅可查看本人数据
- 班主任 / 辅导员：按所带班级、专业或负责范围查看
- 任课教师：查看本人发起和本人课程相关数据
- 教研室：查看本教研室范围数据
- 学院：查看本学院范围数据
- 教务处、科技处、学工处：查看所属业务域全校数据
- 后勤、宿管、资产管理员：查看所属资源和业务域数据
- 系统管理员：拥有系统配置权限，业务数据访问需审计

### 8.3 权限控制要求

- 前端页面与后端接口必须同时校验权限
- 审批节点必须严格按当前待办人和角色控制
- 敏感数据操作需记录操作人、时间、动作和结果

## 11. 验收标准

### 11.1 文档验收标准

- 文档覆盖项目背景、目标、角色、业务、权限、数据、页面和验收标准
- 一期范围和优先级明确
- 高优先级业务具备清晰流程描述

### 11.2 系统功能验收标准

- 管理员可完成基础数据维护
- 不同角色登录后可看到对应菜单和数据
- 流程可完成提交、审批、退回、撤回、办结
- 附件可上传、查看、下载并与业务绑定
- 消息通知能够覆盖关键流程节点

### 11.3 业务验收标准

- 学生请假与销假可形成闭环
- 实习材料可实现提交、审核、补交和汇总
- 异常学生可实现建档、跟踪和归档
- 调课、课程标准评审、课题申报、会议室申请、维修工单等一期业务可跑通主流程

### 11.4 权限与数据验收标准

- 学生不能查看他人数据
- 班主任和辅导员仅能查看负责范围数据
- 学院、职能部门和管理员的权限边界清晰
- 审批操作、配置变更和敏感操作有日志留痕

### 11.5 统计与运行验收标准

- 关键业务能够按组织、时间、状态查询统计
- 超时流程和异常数据能够被识别
- 系统具备继续扩展后续业务的结构基础
## 13. 业务归属修订说明

- 毕业设计开题申请和毕业设计中期检查统一归入学生生命周期业务，由学生发起，指导老师或相关教师审批。
- 教学教务业务仍保留调课、停课、补课、课程标准、教材征订、教案、教学大纲、成绩更正、考试安排和教室借用等教师或教务主导流程。
- 科研成果登记继续归入科研管理，不调整为学生业务。

## 14. 第 7 批需求补充

第 7 批继续扩展校园 OA 高频业务，覆盖后勤用车、班级通知、学生预警、材料补交和公告发布。

- 车辆申请：支持教师提交公务用车申请，记录用车时间、目的地、乘车人数和调度要求，由管理员审批排车。
- 班级通知回执：支持班主任发布班级通知，记录回执截止时间、应回执人数和已回执人数，形成回执统计闭环。
- 学生预警处理：支持班主任登记学生预警，填写触发原因、处理记录和跟进计划，由学工处复核。
- 证明材料补交：支持学生针对被退回或缺失材料发起补交，关联原业务单号并上传补交说明。
- 公告发布：支持教师或管理员提交公告，管理员审核后归档，记录发布范围、计划发布时间和置顶标识。

第 7 批业务仍要求：

- 所有接口必须登录后访问。
- 发起、查看、审批必须经过后端权限校验。
- 附件通过统一附件中心与申请单关联。
- 审批记录不可覆盖，业务状态和流程状态同步维护。

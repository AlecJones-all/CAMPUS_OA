# 数据库设计草案

## 1. 设计目标

数据库设计需满足以下目标：

- 支撑统一用户、角色、组织、权限体系
- 支撑通用流程引擎和多业务复用
- 支撑统一附件、消息、日志管理
- 支撑学生事务、教学教务、科研管理、行政后勤业务扩展
- 支撑统计分析和后续移动端接入

## 2. 项目技术基线

本数据库设计基于当前项目统一技术基线展开：

- 前端：`Vue 3 + Vite + TypeScript + Element Plus`
- 后端：`Spring Boot`
- 数据库：`MySQL`
- 文档：`Markdown`

## 3. 设计原则

- 基础平台表与业务表分离
- 主数据统一管理，避免重复建模
- 通用流程能力独立成域
- 业务状态与流程状态分离
- 审批意见和操作日志不可覆盖
- 附件独立存储并与业务解耦
- 表结构优先满足可维护和可扩展，不为早期优化过度复杂化

## 9. 状态设计建议

### 业务状态

- 草稿
- 已提交
- 审批中
- 已通过
- 已退回
- 已撤回
- 已办结

### 流程状态

- 未启动
- 运行中
- 已完成
- 已终止

说明：

- 业务状态用于表达表单业务本身
- 流程状态用于表达审批流引擎状态
- 两者分离有利于适配更多业务

## 10. 索引设计建议

- 用户、角色、组织编码字段建立唯一索引
- 业务单号建立唯一索引
- 流程实例中 `business_type + business_id` 建组合索引
- 列表高频筛选字段如 `org_id`、`status`、`created_at` 建普通索引
- 消息和待办按用户维度建立索引

## 11. 约束与扩展建议

- 重要主数据使用逻辑删除，不建议物理删除
- 审批记录和日志默认不可删除
- 附件不直接写入业务表，统一关联附件中心
- 后续如需对接对象存储，可在附件表扩展存储类型字段
- 后续如需多租户或多校区，可在组织和业务表增加校区维度

## 12. 一期建议优先建表范围

- 用户、角色、组织、菜单、字典、参数
- 流程定义、流程实例、任务、审批日志
- 附件、消息、操作日志
- 请假、实习材料、异常学生、课程标准评审、课题申报、会议室申请、维修工单

## 14. 数据库验收标准

- 能覆盖一期主流程所需的基础数据与业务数据
- 能满足角色权限和数据权限控制需要
- 能支持待办、已办、流程轨迹和统计查询
- 能支持附件、日志和消息留痕
- 表结构具备后续扩展空间

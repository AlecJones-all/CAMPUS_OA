# AGENTS

本文件用于约束本项目中的协作型开发行为，适用于后续所有文档、代码和数据库相关工作。

## 基本原则

1. 每次开始写代码前，必须先完成计划说明，明确本次目标、影响范围、实现方式和验证方式。
2. 每次修改完成后，必须运行相应检查，至少包括编译、启动、静态检查或最小可运行验证中的一种。
3. 不得删除已有文档。若文档内容需要调整，应采用补充、修订、重命名评估或新增版本的方式处理。
4. 不得擅自更换已确认的技术栈。当前默认技术栈为：
   - 前端主栈：Vue 3 + Vite + TypeScript + Element Plus
   - 前端配套：Pinia + Vue Router
   - 后端主栈：Spring Boot
   - 后端配套：Spring Security + MyBatis-Plus
   - 数据库：MySQL
   - 文档：Markdown
5. 新增模块、表结构、接口前，应先检查现有文档和已有实现，避免重复设计。
6. 所有需求、设计、数据库和接口文档默认使用中文编写。
7. 新增业务模块时，应优先复用统一用户、组织、权限、流程、附件、消息和日志能力，避免单模块重复造轮子。
8. 任何涉及审批流的业务，必须明确发起人、审批节点、退回规则、撤回规则、附件要求和归档规则。
9. 任何涉及角色权限的改动，必须同步更新对应文档，至少包括角色说明、菜单权限或数据权限边界。
10. 未经确认，不得引入重量级工作流引擎、微服务拆分、复杂中间件或与当前阶段不匹配的基础设施。

## 开发前检查清单

- 本次任务目标是否明确
- 是否已有对应文档或已有模块可复用
- 是否会影响数据库结构、接口契约、角色权限或业务流程
- 是否需要同步更新文档
- 是否已定义完成标准和验证方式

## 修改后检查清单

- 后端是否可编译或可启动
- 前端是否可编译或可启动
- 新增接口是否有最小可访问验证
- 数据库脚本是否可执行
- 文档是否同步更新
- 是否误删原有文件或破坏既有结构

## 文档维护要求

- 需求分析文档应持续补充，不以“临时记录”替代正式文档
- 业务流程文档应保持字段完整，至少包含参与角色、流程步骤、表单字段和审批规则
- 数据库设计变更后，应同步修订数据库设计草案
- 若实现内容与既有文档不一致，应先修正文档再继续开发

## Skill 使用规则

当用户要求一次开发 5 个校园 OA 功能、继续批次开发、检查本批次功能实现时，优先使用：

- $campus-oa-batch-5

不要让用户每次重复指定多个 Skill。  
$campus-oa-batch-5 已经包含文档规划、架构设计、后端开发、前端开发、安全检查、代码审查、QA 验收流程。
